# 整数溢出与支付校验漏洞

## 整数溢出基础

- **int类型最大值**: 2147483647（32位有符号整数）
- **溢出行为**: 超过最大值返回0或负数
  - 2147483649 = 1（实际返回 1）

## 场景案例：馒头购买

- 馒头单价：2元
- 购买数量：1073741825个
- 正常总价：2147483650元
- **漏洞利用**：int类型溢出后循环计算结果为2元
- **攻击效果**：仅需支付2元

## 科学计数法绕过

- `1E+2` = 100（1后面跟2个0）
- `1E-1` = 0.1
- **用途**：绕过只能输入整数的校验
- 通过科学计数法可以输入小数

## 金额四舍五入漏洞

- **场景**：充值最小单位为分（0.01元）
- **漏洞原理**：
  - 输入 0.016 元，系统可能只读取到 0.01（四舍五入）
  - 但后端实际存储 0.016 元
- **验证方法**：
  1. 充值 0.016 元（实际扣除 0.02 元）
  2. 再次充值 0.016 元
  3. 若余额显示 0.03 元（而非 0.04 元），则存在漏洞

## 支付校验逻辑漏洞

- **正常逻辑**：校验支付金额是否正确
- **漏洞逻辑**：仅校验是否支付成功，不校验金额
- **利用方式**：构造小金额支付请求，服务器返回成功

## 防护建议

1. 使用更大的整数类型（int64）或使用字符串存储金额
2. 对金额进行严格的前端和后端校验
3. 使用定点数（decimal）而非浮点数存储金额
4. 同时校验支付状态和金额一致性
